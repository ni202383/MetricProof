# MetricProof 仓库工作约束

本文件适用于整个仓库。子目录可以增加更具体的 `AGENTS.md`，但不得放宽这里的产品边界、安全约束或质量要求。

## 产品边界

- MetricProof 是开源、本地优先的 Python CLI。
- 首版仅支持 LaTeX、JSON、YAML、CSV 和本地 Git。
- 诊断表示一致性问题或启发式风险，不得描述为已确认的论文错误、科研不端或科学结论裁决。
- 不调用外部 AI API，不使用 LLM 或嵌入模型。
- 不执行训练代码、TeX、用户脚本或任意表达式。
- 不增加数据库、账号、服务端、Web 应用、付费功能或插件系统。
- 不解析 PDF、Word 或在线 Overleaf 项目。
- 未经明确的新产品决策，不扩大首版范围。

## 工程结构

- 使用 Python 3.12 和 `src` layout。
- 核心包位于 `src/metricproof/`。
- 依赖方向必须保持：

```text
cli → application → domain
adapters → application ports + domain
```

- 核心逻辑不得依赖 CLI 或报告层。
- `domain` 不得导入 Typer、Rich、Jinja2、Pydantic 设置模型或具体适配器。
- 应用服务不得直接实例化文件、Git、LaTeX 或报告适配器。
- 文件系统和 Git 操作必须通过边界接口隔离。
- 规则引擎只接收已准备好的领域对象，不得直接读取任意文件。
- 报告层只渲染 `CheckResult`，不得复制或重新实现规则判断。

## 领域与数值

- 领域模型使用明确类型，不用无约束字典代替核心对象。
- 数值解析和派生计算使用 `Decimal` 或等价的确定性十进制策略。
- 百分数、百分点、相对变化和普通差值必须显式区分。
- 不用格式化字符串相等判断数值一致性。
- Claim ID 不得只依赖绝对行号；必须使用版本化上下文指纹。
- 所有排序必须确定，不依赖随机 UUID、当前时间或遍历偶然顺序。

## 输入、安全与错误

- Windows、Linux 和 macOS 路径统一使用 `pathlib`。
- 进入领域层的文件位置使用项目相对路径，不保存开发机绝对路径。
- 所有配置路径、LaTeX include 和输出路径必须检查项目根目录边界。
- YAML 必须安全加载，禁止任意对象构造。
- 禁止 `eval`、`exec` 和 `shell=True`。
- Git 只能执行只读命令，使用参数列表并设置超时。
- 写入配置、链接和报告文件时优先采用原子替换。
- 不允许静默吞掉解析错误；可恢复错误必须形成结构化诊断。
- 不支持的语法必须受控降级，不得猜测后输出高置信度结论。
- 所有用户可见诊断必须包含 code/rule、位置、证据和严重程度。
- 规则诊断还必须包含 observed、expected、confidence 和 remediation。

## 实现原则

- 首选确定性、可解释算法，不使用 LLM。
- 不为未来功能提前建立复杂抽象。
- 在两个清晰实现足够时，不引入框架化注册表或插件发现。
- 只创建当前阶段真正需要的模块，不批量创建空文件。
- 公共接口应小而明确；格式适配逻辑不得泄漏进领域模型。
- 自动匹配只产生候选和证据，未经用户确认不得写成永久链接。
- 无法唯一迁移 Claim 或链接时必须拒绝自动绑定。

## 测试与质量

- 新功能必须有正例、反例和边界测试。
- 修复缺陷时必须添加回归测试。
- 优先测试领域纯函数、应用服务编排、适配器安全边界和 CLI 退出码。
- 测试不得联网，不依赖用户机器的绝对路径或全局环境。
- 不伪造测试、覆盖率、静态检查或命令执行结果。
- 文档中的命令必须与实际 `pyproject.toml` 保持一致；工程骨架建立前不得声称检查已通过。
- 修改设计语义时同步更新 `SPEC.md`、`ARCHITECTURE.md` 和相关 `docs/`。
- 每个开发阶段完成后更新 `docs/status.md`。

## Git 与外部操作

- 不自动执行 `git commit`、`git push`、创建远程仓库、发布 PyPI 或发布 release。
- 不修改 Git 历史，不运行破坏性 Git 命令。
- 不删除或覆盖用户已有文件，除非任务明确要求且已验证范围。
- 当前本地开发阶段不得创建远程资源。

## 当前阶段执行规则

- 开始工作前阅读 `SPEC.md`、`ARCHITECTURE.md`、本文件和 `docs/status.md`。
- 以现有设计文档为准；发现矛盾时做最小修正并记录理由。
- 不把后续阶段功能提前塞入当前阶段。
- 最终汇报必须区分：已验证事实、未验证设计和已知限制。
